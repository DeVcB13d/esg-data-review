# Data Sources & Formats Researched

## 1. SAP (Fuel and Procurement Data)

**What real-world format I researched:** I analyzed standard SAP Material Management (MM) data structures, specifically utilizing the Kaggle "SAP BigQuery Dataset" which contains raw table dumps (`ekko.csv` for headers, `ekpo.csv` for line items). I also researched standard SAP GUI transaction exports (like ME2N).

**What I learned:** SAP data is highly normalized and natively relies on German database abbreviations. A single purchase order is split across multiple tables. The header (`EKKO`) contains the vendor and dates (`AEDAT`), while the item table (`EKPO`) contains the facility plant code (`WERKS`), the quantity (`MENGE`), and the chaotic units of measure (`MEINS`). The data requires intense mapping before it becomes usable.

**What my sample data looks like and why:** My sample data is a flat, comma-delimited CSV containing pre-joined columns (e.g., `EBELN`, `WERKS`, `MENGE`).
*Why:* Instead of forcing the Django backend to act as a data warehouse ingesting fragmented relational tables, I simulated the reality of a sustainability analyst manually exporting a flat report from the SAP GUI. This allows the prototype to focus on normalizing units and mapping plant codes rather than re-assembling SAP's internal architecture.

**What would break in a real deployment:**

1. **Unmapped Plant Codes:** If the client's SAP system outputs a new `WERKS` code (e.g., "P005") that does not exist in our internal facility lookup dictionary, the ingestion script will fail to attach the emission to a physical location.
2. **Custom Units:** If a facility logs a purchase using a custom SAP unit of measure not handled by our normalizer (e.g., a specific chemical barrel size rather than standard `L` or `KG`), the math will crash or output `NULL`.

## 2. Utility Data (Electricity)

**What real-world format I researched:** I analyzed commercial electricity invoices, specifically utilizing a Commercial-HT2b PDF bill from BESCOM (Bangalore Electricity Supply Company).

**What I learned:** Utility billing is fragmented and localized. The core issues are tariff structures and misaligned dates. The bill spans irregular periods (e.g., Sept 1 to Oct 1) rather than clean calendar quarters. Crucially, commercial bills separate non-emitting "Demand Charges" (measured in KVA) from the actual carbon-emitting "Energy Charges" (measured in kWh).

**What my sample data looks like and why:** The sample data is a raw PDF document of a commercial utility bill.
*Why:* Facilities teams rarely use clean API portals; they manage folders of PDFs. By using this format, I was able to implement an LLM-assisted text extraction method in the backend. It dynamically hunts for the `total_kwh` and service dates, bypassing the need for brittle, layout-dependent regex scrapers that break when a new regional bill is introduced.

**What would break in a real deployment:**

1. **LLM Hallucinations:** The extraction model might confuse the KVA (Demand) number for the kWh (Consumption) number if the bill layout is highly unusual, resulting in wildly inaccurate Scope 2 calculations.
2. **Poor OCR Quality:** If a client uploads a heavily compressed, low-DPI scan of a physical paper bill, the text extraction layer will return garbage characters, causing the JSON parser to fail entirely.

## 3. Corporate Travel (Flights & Hotels)

**What real-world format I researched:** I researched modern Travel Management Company (TMC) webhooks, specifically utilizing the Navan API integration documentation for Expense and Trip data.

**What I learned:** Unlike legacy data, TMC platforms deliver clean, structured JSON payloads. However, they lack the physical variables required for carbon accounting. For example, flight segments often only provide standard IATA airport codes (e.g., `origin: BLR`, `destination: FRA`) rather than the actual flight distance required to multiply against passenger-kilometer emission factors.

**What my sample data looks like and why:** The sample data is a standard JSON payload representing a multi-segment trip (a flight and a hotel stay).
*Why:* It simulates exactly what a production webhook endpoint would receive. I deliberately omitted distance metrics in the JSON to test the backend's ability to cross-reference the airport codes with an internal coordinate dictionary and calculate the distance using the Haversine formula.

**What would break in a real deployment:**

1. **Missing Airport Coordinates:** If an employee flies into a smaller regional airstrip that is not indexed in the backend's latitude/longitude dictionary, the Haversine distance calculation will throw a `KeyError`.
2. **Multi-Leg Layovers:** If the webhook array groups a multi-city trip (e.g., BLR -> DXB -> FRA) as a single segment, calculating the direct point-to-point distance will severely undercount the actual emissions generated by the layover routing.
